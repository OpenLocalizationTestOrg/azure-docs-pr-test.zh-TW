## <a name="disaster-recovery-dr-and-device-failover"></a><span data-ttu-id="e97a1-112">災害復原 (DR) 與裝置容錯移轉</span><span class="sxs-lookup"><span data-stu-id="e97a1-112">Disaster recovery (DR) and device failover</span></span>
<span data-ttu-id="e97a1-113">在災害復原 (DR) 案例中，主要裝置會停止運作。</span><span class="sxs-lookup"><span data-stu-id="e97a1-113">In a disaster recovery (DR) scenario, the primary device stops functioning.</span></span> <span data-ttu-id="e97a1-114">在此情況下，您可以使用主要裝置當做「來源」，並將另一個裝置指定為「目標」，將與失敗裝置相關聯的雲端資料移至另一個裝置。</span><span class="sxs-lookup"><span data-stu-id="e97a1-114">In this situation, you can move the cloud data associated with the failed device to another device by using the primary device as the *source* and specifying another device as the *target*.</span></span> <span data-ttu-id="e97a1-115">您可以選取一或多個磁碟區容器來移轉到目標裝置。</span><span class="sxs-lookup"><span data-stu-id="e97a1-115">You can select one or more volume containers to migrate to the target device.</span></span> <span data-ttu-id="e97a1-116">這個程序就稱為「容錯移轉」 。</span><span class="sxs-lookup"><span data-stu-id="e97a1-116">This process is referred to as the *failover*.</span></span> 

<span data-ttu-id="e97a1-117">在容錯移轉期間，來源裝置的磁碟區容器會變更擁有權，並移轉到目標裝置。</span><span class="sxs-lookup"><span data-stu-id="e97a1-117">During the failover, the volume containers from the source device change ownership and are transferred to the target device.</span></span> <span data-ttu-id="e97a1-118">在磁碟區容器變更擁有權之後，系統會將這些容器從來源裝置中刪除。</span><span class="sxs-lookup"><span data-stu-id="e97a1-118">Once the volume containers change ownership, these are deleted from the source device.</span></span> <span data-ttu-id="e97a1-119">完成刪除之後，目標裝置便可以容錯回復。</span><span class="sxs-lookup"><span data-stu-id="e97a1-119">After the deletion is complete, the target device can then be failed back.</span></span>

<span data-ttu-id="e97a1-120">緊接在 DR 之後，最新的備份通常用來將資料還原至目標裝置。</span><span class="sxs-lookup"><span data-stu-id="e97a1-120">Typically following a DR, the most recent backup is used to restore the data to the target device.</span></span> <span data-ttu-id="e97a1-121">不過，如果相同的磁碟區有多個備份原則，則會挑選具有最大量磁碟區的備份原則，而該原則的最新備份可用於還原目標裝置上的資料。</span><span class="sxs-lookup"><span data-stu-id="e97a1-121">However, if there are multiple backup policies for the same volume, then the backup policy with the largest number of volumes gets picked and the most recent backup from that policy is used to restore the data on the target device.</span></span>

<span data-ttu-id="e97a1-122">例如，如果有兩個備份原則 (一個是預設原則，一個是自訂原則)，defaultPol、customPol 的詳細資料如下︰</span><span class="sxs-lookup"><span data-stu-id="e97a1-122">As an example, if there are two backup policies (one default and one custom) *defaultPol*, *customPol* with the following details:</span></span>

* <span data-ttu-id="e97a1-123">defaultPol︰一個磁碟區 (vol1) 會於每日下午 10:30 開始執行。</span><span class="sxs-lookup"><span data-stu-id="e97a1-123">*defaultPol* : One volume, *vol1*, runs daily starting at 10:30 PM.</span></span>
* <span data-ttu-id="e97a1-124">customPol︰四個磁碟區 (vol1、vol2、vol3、vol4) 會於每日下午 10:00 開始執行。</span><span class="sxs-lookup"><span data-stu-id="e97a1-124">*customPol* : Four volumes, *vol1*, *vol2*, *vol3*, *vol4*, runs daily starting at 10:00 PM.</span></span>

<span data-ttu-id="e97a1-125">在此情況下，將會使用 customPol  ，因為它有更多磁碟區，而我們會針對損毀一致性排定優先順序。</span><span class="sxs-lookup"><span data-stu-id="e97a1-125">In this case, *customPol* will be used as it has more volumes and we prioritize for crash-consistency.</span></span> <span data-ttu-id="e97a1-126">此原則的最新備份可用來還原資料。</span><span class="sxs-lookup"><span data-stu-id="e97a1-126">The most recent backup from this policy is used to restore data.</span></span>
